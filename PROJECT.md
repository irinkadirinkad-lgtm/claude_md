#### API YClients — подтверждено диагностикой
- `POST /api/v1/finance_transactions/{company_id}` — создание операции (работает).
  Поля: expense_id, amount, account_id, client_id, supplier_id, master_id, comment, date ("YYYY-MM-DD HH:MM:SS")
- `GET /api/v1/finance_transactions/{company_id}` — **405 Method Not Allowed**, читать через
  этот эндпоинт нельзя (только создание).
- Авторизация: Authorization: Bearer {partner_token}, User {user_token}
- user_token: POST /api/v1/auth {login, password}
- `GET /api/v1/staff/{company_id}` — 200 OK, список сотрудников (id, name, specialization,
  position{id,title}, fired, status).
- `GET /api/v1/storages/{company_id}` — 200 OK, список складов. У салона их 2: id 637822
  «Основной склад» (for_service — расходники на услуги), id 637823 «Магазин»
  (for_sale — розница/косметика).
- `GET /api/v1/records/{company_id}` — **200 OK** (после того как Ирина расширила права
  техническому пользователю 04.07.2026). Фильтры: `start_date`, `end_date`, `count`. Ключевые
  поля записи: `staff_id` (кто оказал услугу), `created_user_id` (кто оформил визит —
  например, id администратора Элины) — то самое разделение «мастер обслужил» vs
  «администратор записал», которое нужно было для Ввода. `services[]` (title, cost) —
  выручка услуг. `goods_transactions[]` (title, cost, master_id, storage_id, good_id) —
  товары, проданные В РАМКАХ визита. `attendance`/`visit_attendance` — статус посещения
  (1 = пришёл).
- **ВАЖНО про «новый клиент»:** `client.is_new` и `client.success_visits_count` внутри
  записи — это ТЕКУЩЕЕ состояние клиента на момент запроса API, а НЕ историческое состояние
  на момент того визита. Проверено на реальных данных (04.07.2026): у клиента с несколькими
  визитами за 4 месяца `success_visits_count` ОДИНАКОВЫЙ на всех его визитах (например,
  везде «52») — это просто «сколько визитов у него сейчас», не история. Значит: **нельзя
  брать эти поля из старой записи и считать, что они отражают состояние клиента в дату
  записи.** Единственный надёжный способ определить «это был первый визит клиента» —
  считать самим: найти дату самого раннего визита клиента по ВСЕЙ его истории (не только
  за целевой период) и сравнить с датой конкретной записи. Иначе бэкфилл старых месяцев
  задним числом будет давать неверные цифры по новым клиентам.
- `GET /api/v1/schedule/{company_id}/{staff_id}/{start_date}/{end_date}` — **200 OK** (пути
  через `/`, не `?query`). Отдаёт массив по дням: `{date, is_working, slots:[{from,to}]}` —
  это и есть настоящие «Смены», не привязанные к визитам (ровно то, что просила Ирина).
- **Розничные продажи БЕЗ визита — РЕШЕНО, полностью читается задним числом.** Не через
  `records` (там их нет — проверено за 4 месяца, 0 совпадений) и не через `storage_operations`/
  `goods_operations`/`documents` (это всё эндпоинты СОЗДАНИЯ, не чтения — 404 или 400 на
  чтение). Правильный путь нашёлся в официальной документации YClients (не в JS-версии сайта,
  которую я не могла прочитать, а в разделе API-документации, куда Ирина зашла напрямую):
  - **`GET /api/v1/transactions/{company_id}`** — НЕ `finance_transactions` (это другой,
    create-only ресурс). Фильтры: `master_id`, `start_date`, `end_date`, `client_id`,
    `account_id`, `type`, `page`, `count` и др. Возвращает ВСЕ финансовые операции сотрудника
    за период — включая продажи товара БЕЗ визита (`record_id:0, visit_id:0`,
    `expense:{id:7, title:"Продажа товаров"}`) и, что важно, **выручку по услугам тоже**
    (`expense:{id:5, title:"Оказание услуг"}`) — с привязкой к `master_id` в обоих случаях.
    Также встречаются служебные записи типа зарплатных выплат (`expense.id:3`,
    отрицательная сумма) — их отфильтровывать, брать только `expense.id` 5 и 7.
  - Проверено на реальных данных (04.07.2026): по мастеру Луизе за тестовый период отдельно
    просуммировались услуги (`id:5`) и товар (`id:7`), суммы разумные.
  - **Вывод: `transactions` — единственный нужный источник выручки** (и услуг, и товара,
    визит был или нет). Парсить `goods_transactions[]` внутри `records` для товара больше
    не требуется — всё покрывает `transactions`. `records` остаётся нужен только для
    подсчёта визитов/клиентов/нового клиента и разделения мастер (`staff_id`) vs
    администратор (`created_user_id`).
  - Вебхуки (`hooks_settings`) для этой задачи больше НЕ нужны — реальное время не
    требуется, `transactions` и так даёт полную историю назад (можно бэкфилить март-май).

**Двойная роль сотрудника** (пример — Ирина, Арт-Директор + Администратор): ДВЕ отдельные
строки в «Ввод», не пытаться разделить одного человека автоматически по смыслу:
- Как мастер — визиты/выручка/товар по `staff_id` (кто оказал услугу).
- Как администратор — клиенты/новые клиенты/выручка услуг по `created_user_id` (кого
  записала, независимо от того, кто обслуживал). Смены — те же, что и в строке мастера.
- Правило универсальное (`staff_id` vs `created_user_id`), не только для Ирины.
- Товар: и мастер, и администратор МОГУТ иметь личные продажи — считать по каждому

  двухролевого сотрудника (один id под двумя строками в «Настройки») товар задвоится.

**Продажи без сотрудника:** продажи Ирины (в любой роли) идут не в её личную строку, а в

### Планировщик расходов — идёт пересборка (с июля 2026)
Пересборка ведётся по структуре и логике из файла, который Ирина прислала отдельно (не
файл-выгрузка живой таблицы выше — часть вещей в новой версии будет с нуля, часть —
переиспользуется из уже работающей автоматизации в Код.js, см. выше). Работаем и пушим
в `main` этого репозитория напрямую (не в отдельных ветках) — так данные остаются свежими
из сессии в сессию. Подробности решений по пересборке — по ходу работы, актуальное держать
здесь компактно после завершения этапа, не по каждому шагу.
