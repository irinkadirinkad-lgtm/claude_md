## Интеграция каналов продаж — замена МойСклад (YClients ↔ Тильда, потом Ozon)

Большой блок, старт 23.07. Убираем МойСклад — наша интеграция сама становится «связочным звеном».
Этапами: сначала полностью **Тильда**, потом **Ozon** (у Ozon свои цены + правка цен — отдельно, позже).
Согласованная рамка (23.07):

- **Поток A. YClients → Тильда** (источник правды — YClients, склад = ТОЛЬКО магазин/розница;
  сервисный склад расходников не трогаем): остатки + карточки (название, цена). Цена в Тильде =
  как в реальном магазине. Остаток передаём **актуальным числом, а не разницей** (потерянный сигнал
  выровняется на следующей синхронизации — не копит ошибку).
- **Поток B. Тильда → YClients**: заказ на сайте → сразу списание → в YClients создаётся
  **складская операция со списанием, ИТОГ К ОПЛАТЕ = 0 → финансовая операция НЕ создаётся.**
  Онлайн-доход Ирина проводит ОТДЕЛЬНО статьёй дохода. Так каналы не смешиваются: реальная продажа
  в магазине = складская операция С оплатой; онлайн-фулфилмент = складская операция с 0. (Позже это
  же даст финансовому приложению различать каналы — Блок B; финансы подключаем отдельным шагом.)
- **Синхронизация раз в ~5 минут** (Apps Script, бесплатно; мгновенная не нужна).
- **КОНТРОЛЬ — приоритет** (решение Ирины): защита от задвоения и пропусков + **периодическая
  подстраховочная полная сверка** (реконсиляция остатков) + журнал/лог операций для отладки.
  Свои операции метить, чтобы не зациклить (списание из Тильды не улетело обратно как ещё одно).
- **Изоляция:** отдельный проект — свой репозиторий + свой Apps Script + свои ключи, НЕ в куче с
  финансами (связь с финансами — позже).
- Объём ~500+ SKU (не все актуальны). Тильда — полный API-доступ, есть свой каталог товаров.
  Ключ сопоставления YClients↔Тильда — общий артикул (уточняется при изучении).
- **Механизм (подтверждён по докам API, 23.07):**
  • Хаб — наш отдельный Apps Script (бесплатно): триггер каждые 5 мин + doPost-приёмник вебхука.
  • **A (YClients→Тильда):** `GET /goods/{company}` → остаток из `actual_amounts` по складу `for_sale`,
    цена `cost`, ключ сопоставления `article` (артикул). Заливаем в Тильду **фидом CommerceML**
    (import.xml — название/артикул/категория + offers.xml — цена/остаток) на коннектор-URL Тильды
    (HTTP Basic auth), галки «создавать товары / обновлять цены / обновлять остатки». Остаток —
    абсолютным числом (каждый цикл шлём полный остаток = сверка встроена, самовыравнивание).
  • **B (Тильда→YClients):** заказ → вебхук корзины (POST, поля `externalid`=артикул + кол-во) на наш
    doPost → `POST /storage_operations/operation` c `type_id=4` (списание), `cost=0` → остаток вниз,
    финансовая операция НЕ создаётся. `externalid` в Тильде = наш `article` (мы сами наполняем каталог фидом).
  • **Контроль:** полный цикл = реконсиляция; журнал операций; идемпотентность по `tranid` заказа
    (без задвоения при повторной отправке); лист несопоставленных (товары без артикула / заказы без совпадения).
    **Почта Ирине при любой ошибке/непрошедшей операции** — как в финансовой интеграции: копируем
    `sendErrorEmail_` (MailApp, `OWNER_EMAIL_`) + лист «Статус автоматики» (видно и успешные прогоны, не только ошибки).
  • Вебхуков на склад у YClients НЕТ → только polling (5 мин). Tilda публичный API — read-only, товары
    только через CommerceML. **Один CommerceML-коннектор на проект** → при запуске отключить МойСклад от Тильды.
  • ⚠️ Проверить на боевом ДО стройки: (1) что списание `type_id=4` не создаёт финоперацию;
    (2) точные поля вебхука Тильды (снять тестовым заказом).
- Репозиторий: **`tilda-yclients-integracia-magazin`** (имя от Ирины). Приватный. У GitHub-приложения
  НЕТ прав на создание репозиториев (403) → создаёт Ирина вручную на github.com, потом добавляю в сессию.
- «Свои токены» YClients = НЕ новый сотрудник: отдельная КОПИЯ существующих ключей доступа в Script
  Properties нового проекта (изоляция). Доступ тот же, что у финансовой интеграции. Проверить только,
  что у доступа есть права на складские операции (создавать списание) — это галочка прав, не найм.
- СТАТУС: механизм собран, план утверждён. Ждём: (1) Ирина создаёт репозиторий; (2) Тильда —
  CommerceML-коннектор (URL/логин/пароль из каталога); (3) две боевые проверки (списание 0₽, поля вебхука).
